---
title: "SKILL.md Format"
---

Each skill is a directory with a `SKILL.md` at its root. `SKILL.md` has two parts: a **metadata frontmatter** and an **instruction body**.

## Overall Structure

```markdown
---
name: my-skill
display_name: My Skill
description: Short description used by the Agent to decide when to activate
---

## Steps

1. ...
2. ...
```

## Frontmatter Fields

| Field | Required | Description |
|-------|----------|-------------|
| `name` | When creating/registering | Unique identifier. 1–64 characters, letters, digits, hyphens, underscores. Can be omitted when scanning an existing directory — the directory name is used |
| `description` | Strongly recommended | Short description; determines whether the Agent can find this skill |
| `display_name` | No | Display name (used in the UI) |

Other common fields (`license` / `compatibility` / `metadata` / `allowed-tools`) are not read today, but can stay for future compatibility or human reading.

### Writing a Good `description`

`description` is the only thing the Agent uses to "pick a skill" — it shows up in the `<available_skills>` list. A vague description means the skill won't be activated.

```yaml
# ❌ Too vague
description: Helps with PDFs

# ✅ Covers capability, scenario, and trigger keywords
description: Extracts text and tables from PDFs, fills PDF forms, merges multiple PDFs; triggered when the user mentions PDF, forms, or document extraction
```

Writing tips:
- State the **capability** (what it does)
- State the **scenarios** (when to use it)
- Include **trigger keywords** (how users might phrase it)

## Instruction Body

Anything after the frontmatter is Markdown body. After the Agent calls `activate`, the entire body is returned as a Tool Result and injected into context.

Suggested structure (not enforced):

```markdown
## Use Cases
What kinds of requests this skill fits.

## Steps
1. ...
2. ...

## Available Resources
List files under scripts/, templates/, and what each does.

## Output Format
Describe any specific output format here.
```

Keep it short: aim for under 500 lines. Put longer reference material in subdirectories and load it on demand with the `read` tool.

## Full Example

```markdown
---
name: api-tester
display_name: API Testing Tool
description: Tests REST API endpoints, validates responses, and runs performance benchmarks; triggered when the user mentions API testing, endpoint validation, or load testing
---

## Use Cases

When the user needs to fire HTTP requests, validate responses, or load-test a single endpoint.

## Steps

1. Extract method, URL, headers, and body from user input
2. Use `exec` to run `scripts/request.py`
3. Validate status code and response body
4. If performance testing is needed, call `scripts/benchmark.py`
5. Output the result following the template

## Available Resources

- `scripts/request.py` — single HTTP request
- `scripts/benchmark.py` — concurrent load test
- `templates/test_report.md` — report template

## Output Format

Single: URL, status code, response time, response body summary
Batch: success/failure counts, average response time, error list
```

## Directory Layout

Minimal:

```
my-skill/
└── SKILL.md
```

With resources:

```
my-skill/
├── SKILL.md           # Required
├── scripts/           # Optional: executable scripts
├── templates/         # Optional: templates
└── requirements.txt   # Optional: dependencies (Agent can pip install on demand)
```

After activation, the Agent accesses these files through `/workspace/.skills/{skill-name}/`.

## Storage Location

When the sandbox backend is available, skills live in the skill store managed by Box Runtime, default path:

```
<box.local.host_root>/skills/<skill-name>/SKILL.md
```

`box.local.skills_root` defaults to the relative path `skills`, which resolves under `host_root`. When Box is unavailable the skill list is empty; there is no fallback to other directories.
