---
title: "スキルシステム (Skills)"
---

Skills（スキル）は LangBot の動的スキルシステムで、AI Agent が Tool Call メカニズムを通じてオンデマンドで特定のタスクを起動・実行できるようにします。Skills を通じて、Agent が必要な時に専門分野の知識やツールを自動的に読み込むことができます。

## Skills とは？

Skills は軽量な拡張メカニズムで、Agent に以下の機能を提供します：

- Tool Call でオンデマンドに専門分野の知識を読み込む
- 特定タスクのワークフローを実行する
- 定義済みのコードとリソースにアクセスする
- 複数のスキルを動的に組み合わせる

プラグインとは異なり、Skills はより軽量で柔軟、特定タスクのワークフローや命令の定義に適しています。

## 設計理念

### 業界の実践

Skills システムの設計は、業界の主要な Agent 拡張パターンを参考にしています：

| システム | メカニズム | 特徴 |
|----------|------------|------|
| **Claude Code** | Skills + Commands | Tool Call による起動、Tool Result による注入（KV Cache を保護） |
| **OpenAI GPTs** | Actions + Knowledge | ツール呼び出しと知識注入を分離 |
| **Cursor** | Rules + Tools | ルールシステムとツールシステムを独立設定 |

### Claude Code からの啓示

Claude Code の Skills システムには重要な設計決定があります：**スキルコンテンツは Tool Result を通じて注入され、System Prompt を直接変更しません**。

この設計の理由：

1. **KV Cache の保護**: System Prompt を直接変更すると、以前の KV Cache が無効になり、毎回再計算が必要になる
2. **オンデマンド読み込み**: Agent が `activate` ツールを能動的に呼び出した時だけスキルが注入され、不必要なトークン消費を削減
3. **増分コンテキスト**: Tool Result は増分コンテンツで、既存のキャッシュに影響しない

### LangBot の設計

LangBot は Claude Code の設計パターンを完全に採用しています：

```
┌─────────────────────────────────────────┐
│         Agent (LLM)                      │
│                                          │
│  ユーザーリクエスト: "このPython          │
│  プロジェクトを分析して"                   │
│                                          │
│  判断: skill: python-analyzer にマッチ    │
│                                          │
│  呼び出し: activate(skill_name="python-..│
│                                          │
└───────────────────┬─────────────────────┘
                    │ Tool Call
                    ▼
┌─────────────────────────────────────────┐
│           activate ツール                 │
│                                          │
│  戻り値: Tool Result                     │
│  - SKILL.md 内容                         │
│  - マウントパス情報                       │
│                                          │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│   Agent が継続実行（スキルコンテキスト付き）│
│                                          │
│  - exec/read/write/edit でスキルパッケージ│
│    を操作                                │
│  - スキル命令に従ってタスクを実行          │
│  - activate を複数回呼び出し可能          │
│                                          │
└─────────────────────────────────────────┘
```

### なぜこの設計なのか？

**1. KV Cache 保護**

スキル命令は Tool Result として増分コンテンツで返され、既存の KV Cache を無効化しません。

**2. プログレッシブディスクロージャー**

- `activate` ツール説明にはスキルメタデータ（名前、説明）のみ表示
- 完全な命令は起動後に Tool Result で取得
- Agent は必要に応じて起動するか選択可能

**3. 統一されたインタラクションパターン**

- 他のツール呼び出しと一貫したセマンティクス
- Agent 設計パターンにより適合
- Function Calling をサポートするモデルに友好的

**4. サンドボックスバックエンドが必要**

`activate` と `register_skill` ツールはサンドボックスバックエンドが利用可能である必要があります：
- スキルパッケージは `/workspace/.skills/{skill-name}/` にマウント
- Agent は `exec`/`read`/`write`/`edit`/`glob`/`grep` でスキルファイルを操作

## Skill ファイル構造

各 Skill は `SKILL.md` ファイルを含むディレクトリです。サンドボックスバックエンドが利用可能な場合、Skill パッケージは Box Runtime が管理し、`box.local.skills_root` に保存されます。デフォルトは `<box.local.host_root>/skills` です：

```
<box.local.host_root>/skills/
├── my-skill/
│   ├── SKILL.md           # 必須: スキル定義ファイル
│   ├── scripts/           # オプション: スクリプトファイル
│   │   └── analyze.py
│   └── templates/         # オプション: テンプレートファイル
│       └── report.md
└── another-skill/
    └── SKILL.md
```

<Note>
Box Runtime が利用できないローカル開発時のみ、従来の `data/skills/` ディレクトリにフォールバックします。
</Note>

### SKILL.md フォーマット

YAML frontmatter を使用してメタデータを定義：

```markdown
---
name: python-analyzer
display_name: Python プロジェクトアナライザー
description: Python プロジェクトの構造、コード品質、依存関係を分析
---

## 使用場面

ユーザーが Python プロジェクトを分析する必要がある場合、以下の手順で実行：

1. `read` ツールでプロジェクト構造を読み込む
2. `exec` で分析スクリプトを実行
3. 分析レポートを生成

## 利用可能なスクリプト

- `scripts/analyze.py`: メイン分析スクリプト
- `scripts/dependency_check.py`: 依存関係チェック

## 出力フォーマット

分析レポートに含める項目：
- プロジェクト構造の概要
- コード品質スコア
- 依存関係グラフ
```

## 起動メカニズム

### Tool Call による起動

Agent は `activate` ツールを呼び出してスキルを起動：

```json
{
  "skill_name": "python-analyzer"
}
```

### 起動後の効果

1. **Tool Result 注入**: SKILL.md 内容が Tool Result として返され、コンテキストに注入
2. **パスマウント**: スキルパッケージが `/workspace/.skills/{skill-name}/` にマウント
3. **ツールアクセス**: Agent は `exec`/`read`/`write`/`edit`/`glob`/`grep` でスキルファイルを操作可能

### 戻り内容の例

```
<command-message>The "python-analyzer" skill is activated</command-message>
<skill-activation>
<skill-name>python-analyzer</skill-name>
<mount-path>/workspace/.skills/python-analyzer</mount-path>
<package-root>/path/to/skill</package-root>

## Instructions
...SKILL.md 内容...

## Runtime Context
The skill package is mounted at /workspace/.skills/python-analyzer. Use the standard tools to interact with it:
- Use `read` to inspect files under /workspace/.skills/python-analyzer
- Use `exec` with workdir set to /workspace/.skills/python-analyzer to run commands
- Use `write` and `edit` on that path when the instructions require updating files
</skill-activation>
```

## 組み込みツール

Skills システムは Agent が使用する2つのツールを提供：

| ツール | 機能 | サンドボックス必要 |
|--------|------|-------------------|
| `activate` | スキルを起動 | 必要 |
| `register_skill` | サンドボックスディレクトリからスキルを登録 | 必要 |

<Note>
これらのツールはサンドボックスバックエンド（Docker/Nsjail/E2B）が利用可能である必要があります。バックエンドがない場合、ツールは Agent に提供されません。
</Note>

## スキルタイプ

### プロンプトのみのスキル

命令のみを含み、追加ファイル不要：

```markdown
---
name: code-reviewer
display_name: コードレビューアシスタント
description: コードをレビューし、改善提案を提供
---

## レビュー手順

1. コード構造とスタイルを分析
2. 潜在的なバグとセキュリティ問題をチェック
3. 改善提案を提供

## レビュー基準

- コード可読性
- パフォーマンス最適化
- セキュリティベストプラクティス
```

### ファイルパッケージスキル

スクリプト、テンプレートなどのリソースファイルを含む：

```
data-analyzer/
├── SKILL.md
├── scripts/
│   ├── analyze.py
│   └── visualize.py
├── templates/
│   └── report.md
└── requirements.txt
```

## クイックスタート

### 1. 最初のスキルを作成

WebUI の Skills ページで作成するか、サンドボックス `/workspace` に Skill パッケージを準備して `register_skill` で登録します。手動で準備する場合：

```bash
mkdir -p /workspace/my-first-skill
```

`SKILL.md` を作成：

```markdown
---
name: my-first-skill
display_name: 私の最初のスキル
description: シンプルなサンプルスキル
---

## タスク説明

これは基本的なスキル機能をデモンストレーションするサンプルスキルです。

## 実行手順

1. ユーザーリクエストを理解
2. 対応する操作を実行
3. 結果を返す
```

### 2. 登録して起動をテスト

`register_skill(path="/workspace/my-first-skill")` を呼び出してスキルを登録します。サンドボックスバックエンドが利用可能であることを確認し、スキルの説明にマッチするリクエストを Agent に送信します。Agent は自動的に `activate` ツールを呼び出してスキルを起動します。

### 3. パイプラインで Skill を有効化

Skills はパイプライン拡張です。作成、インポート、登録した後、パイプライン拡張設定で許可する必要があります。

- すべての Skills を有効化する
- または指定した Skills だけをバインドする

バインドは、Agent から見える Skills を制御するだけです。完全な `SKILL.md` 指示は事前にコンテキストへ入りません。Agent が `activate` を呼び出した後に Tool Result として返されます。

<Note>
Skills は現在、組み込みの Local Agent 向けで、Tool Call と sandbox backend に依存します。外部 Runner では、それぞれのツールとコンテキスト機構を使用します。
</Note>

## 新しいスキルの作成

### 方法1: WebUI またはインポートを使用

WebUI の Skills ページでは、作成、編集、GitHub からのインポート、zip パッケージのアップロードができます。サンドボックスバックエンドが利用可能な場合、これらのスキルは Box Runtime 管理の skill store に書き込まれます。

### 方法2: Agent にサンドボックス内で作成させる

1. サンドボックス `/workspace` 下にスキルディレクトリを作成
2. `SKILL.md` と関連ファイルを記述
3. `register_skill(path="/workspace/my-skill")` を呼び出して登録

## ベストプラクティス

### 1. スキル命名

- 小文字、数字、ハイフン、アンダースコアを使用
- 名前は簡潔で説明的に
- 例: `python-analyzer`、`data-visualizer`

### 2. 説明の記述

- スキルの用途と適用場面を明確に記述
- LLM がいつ起動するか正しく判断できるよう支援
- 説明は `activate` ツールの `<available_skills>` リストに表示

### 3. 命令設計

- 明確な実行手順を提供
- 利用可能なリソースとツールを説明
- 出力フォーマットを定義

### 4. リソース整理

- スクリプトは `scripts/` ディレクトリに
- テンプレートは `templates/` ディレクトリに
- 構造を明確に保つ

## よくある問題

| 症状 | 確認すること |
|------|--------------|
| Skill リストが空 | Box Runtime が利用可能か、`box.local.skills_root` が期待するディレクトリを指しているか |
| Agent が `activate` を呼び出さない | Skill の description が明確か、パイプラインでバインドされているか、モデルが function calling をサポートするか |
| 有効化後にファイルが見つからない | ファイルが Skill パッケージ内にあるか、`/workspace/.skills/{skill-name}/` パスを使っているか |
| `register_skill` が失敗する | 対象パスは `/workspace` 配下である必要があり、ディレクトリまたは 2 階層以内のサブディレクトリに `SKILL.md` が必要です |
