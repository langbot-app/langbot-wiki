---
title: "スキルシステム (Skills)"
---

Skills（スキル）は LangBot の動的スキルシステムで、AI Agent がオンデマンドで特定のタスクをアクティブ化して実行できるようにします。Skills を通じて、Agent が必要な時に専門分野の知識やツールを自動的に読み込むことができます。

## Skills とは？

Skills は軽量な拡張メカニズムで、Agent に以下の機能を提供します：

- オンデマンドで専門分野の知識を読み込む
- 特定タスクのワークフローを実行する
- 定義済みのコードとリソースにアクセスする
- 複数のスキルを動的に組み合わせる

プラグインとは異なり、Skills はより軽量で柔軟、特定タスクのワークフローや命令の定義に適しています。

## 設計理念

### 業界の実践

Skills システムの設計は、業界の主要な Agent 拡張パターンを参考にしています：

| システム | メカニズム | 特徴 |
|----------|------------|------|
| **Claude Code** | Skills + Commands | Tool Call による起動、Tool Result による注入（KV Cache を保護） |
| **OpenAI GPTs** | Actions + Knowledge | ツール呼び出しと知識注入を分離 |
| **Cursor** | Rules + Tools | ルールシステムとツールシステムを独立設定 |

### Claude Code からの啓示

Claude Code の Skills システムには重要な設計決定があります：**スキルコンテンツは Tool Result を通じて注入され、System Prompt を直接変更しません**。

この設計の理由：

1. **KV Cache の保護**: System Prompt を直接変更すると、以前の KV Cache が無効になり、毎回再計算が必要になる
2. **オンデマンド読み込み**: トリガー条件に一致した時だけスキルが注入され、不必要なトークン消費を削減
3. **増分コンテキスト**: Tool Result は増分コンテンツで、既存のキャッシュに影響しない

### LangBot の設計

LangBot は**レイヤードアーキテクチャ**を採用し、Skills 管理と実行を分離しています：

```
┌─────────────────────────────────────────┐
│         Skills 管理レイヤー               │
│  create_skill, list_skills, update...    │
│  - サンドボックス非依存                   │
│  - 純粋なメタデータ操作                   │
│  - プロンプトのみのスキルに適用            │
└───────────────────┬─────────────────────┘
                    │ 必要時
┌───────────────────▼─────────────────────┐
│      import_skill_from_directory         │
│      - ワークスペースアクセスにサンドボックス必要 │
│      - スクリプト含むスキルパッケージをインポート │
└───────────────────┬─────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│         サンドボックス実行レイヤー         │
│  exec, read, write, edit                 │
│  - スキル内のスクリプトを実行              │
│  - スキルパッケージファイルを操作          │
└─────────────────────────────────────────┘
```

### なぜこの設計なのか？

**1. 利用のハードルを下げる**

プロンプトのみのスキル（命令のみ、スクリプトなし）はサンドボックス設定なしで作成・使用可能：

```yaml
# このスキルはサンドボックス不要
name: code-reviewer
instructions: |
  コードレビュー時に注目：
  - コードスタイルの一貫性
  - 潜在的なセキュリティ脆弱性
  - パフォーマンス最適化の提案
```

**2. 関心事の分離**

| 操作タイプ | サンドボックス必要 | 理由 |
|------------|-------------------|------|
| プロンプトのみのスキル作成/更新 | 不要 | メタデータのみ、コード実行なし |
| スクリプト含むスキルのインポート | 必要 | ファイルシステムアクセス必要 |
| スキル内スクリプトの実行 | 必要 | 分離実行環境必要 |

**3. セキュリティ**

- 単純な CRUD 操作は Agent にファイルシステムアクセス権限を与える必要がない
- 必要な時だけサンドボックス分離を有効化

**4. パフォーマンス**

- 直接的なメモリ/データベース操作はサンドボックス操作より高速
- 不要なサンドボックス起動オーバーヘッドを削減

## 仕組み

```
┌─────────────────────────────────────────────────────────────┐
│                     Agent (LLM)                              │
│                                                              │
│  ユーザーリクエスト: "このPythonプロジェクトを分析して"         │
│                                                              │
│  判断: skill: python-analyzer にマッチ                        │
│                                                              │
│  出力: [ACTIVATE_SKILL: python-analyzer]                     │
│                                                              │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   Skill Activation                           │
│                                                              │
│  1. 起動マーカーを検出（正規表現マッチ）                        │
│  2. マーカーを削除、スキル命令を System Message として注入      │
│  3. スキルパッケージを /workspace/.skills/{skill-name}/ にマウント │
│                                                              │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│              Agent が継続実行（スキルコンテキスト付き）          │
│                                                              │
│  - exec/read/write/edit でスキルパッケージを操作               │
│  - スキル命令に従ってタスクを実行                               │
│  - 複数スキルを同時に起動して組み合わせ可能                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

<Note>
**現在の実装**: LangBot はテキストマーカー `[ACTIVATE_SKILL:]` + System Message 注入メカニズムを使用しています。

**TODO 将来の最適化**: Claude Code の設計に従い、Tool Call / Tool Result メカニズムへの移行を検討：
- メリット: KV Cache を保護、Tool Result は増分コンテンツでキャッシュを無効化しない
- メリット: ツール呼び出しのセマンティクスと一貫性、Agent 設計パターンにより適合
- 現在の設計の理由: Function Calling をサポートしないモデルとの互換性、マーカーメカニズムは任意の LLM で動作
</Note>

## Skill ファイル構造

各 Skill は `SKILL.md` ファイルを含むディレクトリです：

```
data/skills/
├── my-skill/
│   ├── SKILL.md           # 必須: スキル定義ファイル
│   ├── scripts/           # オプション: スクリプトファイル
│   │   └── analyze.py
│   └── templates/         # オプション: テンプレートファイル
│       └── report.md
└── another-skill/
    └── SKILL.md
```

### SKILL.md フォーマット

YAML frontmatter を使用してメタデータを定義：

```markdown
---
name: python-analyzer
display_name: Python プロジェクトアナライザー
description: Python プロジェクトの構造、コード品質、依存関係を分析
---

## 使用場面

ユーザーが Python プロジェクトを分析する必要がある場合、以下の手順で実行：

1. `read` ツールでプロジェクト構造を読み込む
2. `exec` で分析スクリプトを実行
3. 分析レポートを生成

## 利用可能なスクリプト

- `scripts/analyze.py`: メイン分析スクリプト
- `scripts/dependency_check.py`: 依存関係チェック

## 出力フォーマット

分析レポートに含める項目：
- プロジェクト構造の概要
- コード品質スコア
- 依存関係グラフ
```

## 起動メカニズム

### 自動起動

LLM がユーザーリクエストがスキルにマッチすると判断した場合、レスポンスの先頭に起動マーカーを出力：

```
[ACTIVATE_SKILL: python-analyzer]
```

複数のスキルを同時に起動可能：

```
[ACTIVATE_SKILL: python-analyzer]
[ACTIVATE_SKILL: report-generator]
```

### 起動後の効果

1. **命令注入**: スキルの `instructions` が System Prompt に注入される
2. **パスマウント**: スキルパッケージが `/workspace/.skills/{skill-name}/` にマウントされる
3. **ツールアクセス**: Agent は `exec`/`read`/`write`/`edit` でスキルファイルを操作可能

## 組み込みツール

Skills システムは Agent が使用する以下のツールを提供：

| ツール | 機能 | サンドボックス必要 |
|--------|------|-------------------|
| `create_skill` | 新しい管理スキルを作成 | 不要 |
| `list_skills` | すべての管理スキルを一覧 | 不要 |
| `get_skill` | 単一スキルの詳細を取得 | 不要 |
| `update_skill` | 既存スキルを更新 | 不要 |
| `delete_skill` | スキルを削除 | 不要 |
| `import_skill_from_directory` | ワークスペースからスキルをインポート | 必要 |
| `reload_skills` | スキルを再読み込み | 不要 |

<Note>
`create_skill`、`list_skills`、`get_skill`、`update_skill`、`delete_skill` はサンドボックスに依存せず、プロンプトのみのスキルを直接操作可能。`import_skill_from_directory` はワークスペースディレクトリにアクセスするためにサンドボックスが必要。
</Note>

## スキルタイプ

### プロンプトのみのスキル

命令のみを含み、追加ファイル不要：

```markdown
---
name: code-reviewer
display_name: コードレビューアシスタント
description: コードをレビューし、改善提案を提供
---

## レビュー手順

1. コード構造とスタイルを分析
2. 潜在的なバグとセキュリティ問題をチェック
3. 改善提案を提供

## レビュー基準

- コード可読性
- パフォーマンス最適化
- セキュリティベストプラクティス
```

### ファイルパッケージスキル

スクリプト、テンプレートなどのリソースファイルを含む：

```
data-analyzer/
├── SKILL.md
├── scripts/
│   ├── analyze.py
│   └── visualize.py
├── templates/
│   └── report.md
└── requirements.txt
```

## Pipeline へのバインド

Pipeline 設定でスキルをバインドして、その Pipeline で利用可能にできます：

```yaml
# Pipeline 設定
ai:
  local-agent:
    bound_skills:
      - python-analyzer
      - report-generator
```

バインドされたスキルは、その Pipeline の Agent で選択可能なスキルとして表示されます。

## クイックスタート

### 1. 最初のスキルを作成

`data/skills/` ディレクトリにスキルを作成：

```bash
mkdir -p data/skills/my-first-skill
```

`SKILL.md` を作成：

```markdown
---
name: my-first-skill
display_name: 私の最初のスキル
description: シンプルなサンプルスキル
---

## タスク説明

これは基本的なスキル機能をデモンストレーションするサンプルスキルです。

## 実行手順

1. ユーザーリクエストを理解
2. 対応する操作を実行
3. 結果を返す
```

### 2. Pipeline でバインド

Pipeline の AI 設定に追加：

```yaml
bound_skills:
  - my-first-skill
```

### 3. 起動をテスト

スキルの説明にマッチするリクエストを Agent に送信し、スキルが正しく起動されるか確認。

## スキル管理

### 既存スキルの確認

Agent は `list_skills` で全管理スキルを確認：

```
list_skills()
```

戻り値：

```json
{
  "skills": [
    {
      "name": "python-analyzer",
      "display_name": "Python プロジェクトアナライザー",
      "description": "Python プロジェクトを分析..."
    }
  ],
  "count": 1
}
```

### スキルの動的作成

Agent は `create_skill` でスキルを動的に作成：

```
create_skill(
  name="code-formatter",
  display_name="コードフォーマッター",
  description="様々なプログラミング言語のコードをフォーマット",
  instructions="## 使用方法\n..."
)
```

### ワークスペースからインポート

スキルにスクリプトファイルが含まれる場合、サンドボックスで準備してからインポート：

```
# 1. サンドボックスでスキルディレクトリを作成
exec("mkdir -p /workspace/my-skill/scripts")

# 2. スキルファイルを書き込み
write("/workspace/my-skill/SKILL.md", "---\nname: my-skill\n...")
write("/workspace/my-skill/scripts/main.py", "# メインスクリプト")

# 3. 管理スキルとしてインポート
import_skill_from_directory(path="/workspace/my-skill")
```

## ベストプラクティス

### 1. スキル命名

- 小文字、数字、ハイフン、アンダースコアを使用
- 名前は簡潔で説明的に
- 例: `python-analyzer`、`data-visualizer`

### 2. 説明の記述

- スキルの用途と適用場面を明確に記述
- LLM がいつ起動するか正しく判断できるよう支援

### 3. 命令設計

- 明確な実行手順を提供
- 利用可能なリソースとツールを説明
- 出力フォーマットを定義

### 4. リソース整理

- スクリプトは `scripts/` ディレクトリに
- テンプレートは `templates/` ディレクトリに
- 構造を明確に保つ
