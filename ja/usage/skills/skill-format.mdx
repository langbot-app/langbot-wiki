---
title: "SKILL.md フォーマットリファレンス"
---

SKILL.md はスキルを定義するコアファイルで、YAML frontmatter フォーマットを使用します。

<Note>
LangBot の SKILL.md フォーマットは [Agent Skills オープン標準](https://agentskills.io/specification)に従っています。これは Anthropic が主導するクロスプラットフォームのスキル定義仕様です。
</Note>

## ファイルフォーマット

```markdown
---
# YAML Frontmatter (メタデータ)
name: skill-name
description: スキルの説明
---

# Markdown 本文 (命令)
## 使用場面
...
```

## Frontmatter フィールド

### 必須フィールド

| フィールド | 型 | 説明 |
|------------|-----|------|
| `name` | string | スキルの一意識別子 |
| `description` | string | LLM が起動を判断するための簡潔な説明 |

<Note>
Agent Skills 標準では `description` は必須フィールドです。LangBot は現在、後方互換性のために省略を許可していますが、常に提供することを推奨します。
</Note>

### フィールド詳細

#### `name`

スキルの一意識別子。起動と参照に使用されます。

```yaml
name: python-analyzer
```

命名規則（Agent Skills 標準に従う）：
- 1-64 文字
- 小文字 (a-z)、数字 (0-9)、ハイフン (`-`) のみ
- 文字で始まる必要があり、ハイフンで開始または終了できない
- 連続するハイフン (`--`) を含むことはできない
- スキルディレクトリ名と一致することが推奨

#### `description`

スキルの簡潔な説明。LLM はこれを使用してスキルを起動するかどうかを判断します。

```yaml
description: Python プロジェクトの構造、コード品質、依存関係を分析
```

記述ガイドライン：
- 必須、最大 1024 文字
- スキルの機能と目的を記述
- 適用場面とトリガーキーワードを説明
- LLM がいつ起動するか判断できるよう支援

**良い例:**
```yaml
description: PDF ファイルからテキストとテーブルを抽出し、PDF フォームに入力し、複数の PDF を結合します。PDF ドキュメントを扱う場合や、ユーザーが PDF、フォーム、ドキュメント抽出に言及した場合に使用します。
```

**悪い例:**
```yaml
description: PDF を処理します。
```

### オプションフィールド

| フィールド | 型 | 説明 |
|------------|-----|------|
| `license` | string | ライセンス名または参照 |
| `compatibility` | string | 環境要件（製品、システムパッケージ、ネットワークアクセス等） |
| `metadata` | object | カスタムキーバリューメタデータ |
| `allowed-tools` | string | 事前承認済みツールリスト（実験的） |

#### `metadata`

追加プロパティを格納するためのカスタムキーバリューメタデータ。

```yaml
metadata:
  display_name: Python プロジェクトアナライザー
  author: my-org
  version: "1.0"
```

<Note>
LangBot は現在、便宜上トップレベルの `display_name` フィールドをサポートしています。将来のバージョンでは標準の `metadata.display_name` フォーマットに移行し、両方が互換性を保ちます。
</Note>

## 命令本文

frontmatter 以降の内容はスキルの命令で、Agent の System Prompt に注入されます。

### 推奨構造

```markdown
---
name: data-analyzer
display_name: データアナライザー
description: データファイルを分析しレポートを生成
---

## 使用場面

ユーザーが CSV、JSON などのデータファイルを分析する必要がある場合に使用。

## 実行手順

1. `read` ツールでデータファイルを読み込む
2. `exec` で分析スクリプトを実行
3. 構造化レポートを生成

## 利用可能なリソース

- `scripts/analyze.py`: データ分析スクリプト
- `templates/report.md`: レポートテンプレート

## 出力フォーマット

分析レポートに含める項目：
- データ概要
- 統計サマリー
- 可視化提案

## 注意事項

- 大きなファイルはバッチ処理
- データプライバシーに注意
```

### 命令記述のベストプラクティス

#### 1. 使用場面を明確化

```markdown
## 使用場面

以下のタスクをリクエストされた場合にこのスキルを起動：
- Python プロジェクト構造の分析
- コード品質のチェック
- 依存関係レポートの生成

適していない場面：
- 単純なコードスニペットの実行
- Python 以外のプロジェクト分析
```

#### 2. 明確な実行手順を提供

```markdown
## 実行手順

1. **ターゲット確認**: 分析するプロジェクトパスを明確化
2. **構造スキャン**: `exec` で `find . -type f -name "*.py"` を実行
3. **コード分析**: `scripts/analyze.py` を実行
4. **レポート生成**: テンプレートに基づいて結果を出力
```

#### 3. 利用可能なツールとリソースを説明

```markdown
## 利用可能なツール

| ツール | 目的 |
|--------|------|
| `read` | プロジェクトファイルの読み込み |
| `exec` | 分析スクリプトの実行 |
| `write` | レポートファイルの出力 |

## 利用可能なスクリプト

- `scripts/analyze.py`: メイン分析スクリプト
- `scripts/complexity.py`: 複雑度分析
```

#### 4. 出力フォーマットを定義

```markdown
## 出力フォーマット

```json
{
  "project": "プロジェクト名",
  "files": 100,
  "lines": 5000,
  "quality_score": 85,
  "recommendations": ["..."]
}
```
```

## 完全な例

### プロンプトのみのスキル

```markdown
---
name: code-reviewer
display_name: コードレビューアシスタント
description: コードをレビューし改善提案を提供、複数のプログラミング言語をサポート
---

## 使用場面

ユーザーがコードレビュー、コード検査、コード改善提案をリクエストした場合に起動。

## レビュー観点

### 1. コード品質
- 可読性と命名規則
- コード構造と組織
- コメントとドキュメント

### 2. 潜在的な問題
- バグとロジックエラー
- セキュリティ脆弱性
- パフォーマンス問題

### 3. ベストプラクティス
- 言語固有のベストプラクティス
- デザインパターンの適用
- テストカバレッジ

## レビュープロセス

1. コード構造とスタイルを分析
2. 潜在的な問題を特定
3. 具体的な改善提案を提供
4. 優先順位をつけて推奨事項を提示

## 出力フォーマット

### 問題レポート
- **重大度**: 高/中/低
- **場所**: ファイル名:行番号
- **説明**: 問題の説明
- **提案**: 改善提案

### サマリー
- 総合スコア (1-10)
- 主な発見事項
- 優先改善項目
```

### ファイルパッケージスキル

```markdown
---
name: api-tester
display_name: API テストツール
description: REST API エンドポイントをテスト、レスポンスとパフォーマンスを検証
---

## 使用場面

ユーザーが API のテスト、インターフェースの検証、API デバッグを必要とする場合に起動。

## 利用可能なリソース

### スクリプト
- `scripts/request.py`: HTTP リクエストツール
- `scripts/benchmark.py`: パフォーマンスベンチマーク

### テンプレート
- `templates/test_report.md`: テストレポートテンプレート

## 実行手順

1. **リクエスト解析**: ユーザー入力から API 情報を抽出
2. **リクエスト実行**: `exec` で `scripts/request.py` を実行
3. **レスポンス検証**: ステータスコードとレスポンスボディをチェック
4. **レポート生成**: テスト結果を出力

## コマンド例

```bash
# 基本リクエストテスト
python scripts/request.py GET https://api.example.com/users

# 認証付きリクエスト
python scripts/request.py POST https://api.example.com/data \
  --header "Authorization: Bearer token" \
  --body '{"key": "value"}'

# パフォーマンステスト
python scripts/benchmark.py https://api.example.com/endpoint --requests 100
```

## 出力フォーマット

### 単一テスト
- リクエスト URL
- レスポンスステータスコード
- レスポンス時間
- レスポンスボディサマリー

### バッチテスト
- 成功/失敗数
- 平均レスポンス時間
- エラー詳細リスト
```

## ファイル配置

スキルファイルは以下の場所に配置：

```
data/skills/
├── {skill-name}/
│   └── SKILL.md
```

LangBot は起動時に `data/skills/` ディレクトリ内のすべてのスキルを自動的にスキャンします。

## マルチファイルスキル

複数のファイルを含むスキルの場合：

```
data/skills/
└── my-skill/
    ├── SKILL.md           # 必須
    ├── scripts/           # オプション: スクリプト
    │   ├── main.py
    │   └── utils.py
    ├── templates/         # オプション: テンプレート
    │   └── output.md
    └── requirements.txt   # オプション: 依存関係
```

Agent は `/workspace/.skills/{skill-name}/` 経由でこれらのファイルにアクセスできます。

## Agent Skills 標準との関係

LangBot の SKILL.md フォーマットは [Agent Skills オープン標準](https://agentskills.io/specification)に従っています。これは Anthropic が主導するクロスプラットフォームのスキル定義仕様です。

### 標準ディレクトリ構造

```
skill-name/
├── SKILL.md          # 必須: メタデータ + 命令
├── scripts/          # オプション: 実行可能コード
├── references/       # オプション: リファレンスドキュメント
└── assets/           # オプション: 静的リソース
```

### プログレッシブローディングモデル

Agent Skills 標準はプログレッシブローディングを採用し、コンテキスト使用を最適化：

| ステージ | 読み込み内容 | 推奨サイズ | タイミング |
|----------|--------------|------------|------------|
| 発見 | `name` + `description` | ~100 tokens | Agent 起動時 |
| 起動 | SKILL.md 本文 | < 5000 tokens | スキル起動時 |
| 実行 | scripts/、references/、assets/ | オンデマンド | タスク実行時 |

**ベストプラクティス**: SKILL.md 本文は 500 行以内に保ち、詳細な内容は references/ ディレクトリに配置。

### LangBot 互換性ノート

LangBot は現在、便宜上トップレベルの `display_name` フィールドをサポートしていますが、これは Agent Skills 標準フィールドではありません。

将来のバージョンでは：
- `metadata.display_name` 標準フォーマットをサポート
- トップレベルの `display_name` と `metadata.display_name` は両方互換
- 新しいスキルには標準フォーマットの使用を推奨：

```yaml
---
name: my-skill
description: スキルの説明
metadata:
  display_name: 私のスキル
---
```
