---
title: "コードサンドボックス (Sandbox)"
---

LangBotのコードサンドボックス機能は、AIエージェントが分離された環境で安全にコマンドを実行し、ファイルを読み書きし、MCPサーバーを実行することを可能にします。これにより、AIボットに真のコード実行能力が与えられます。

## コードサンドボックスとは？

コードサンドボックスは、エージェントが以下を行える分離された実行環境です：

- シェルコマンドの実行（`python`、`npm`、`curl`など）
- ファイルの作成、読み取り、変更
- 長時間実行されるプロセス（MCPサーバーなど）の実行
- ネットワークリソースへのアクセス（設定に基づく）

一方で、サンドボックスはセキュリティ分離を提供します：

- 実行プロセスはホストシステムに影響しません
- リソース使用量が制限されます（CPU、メモリ、プロセス数）
- ファイルアクセス範囲が制御されます

## 使用例

| シナリオ | 説明 |
|----------|------|
| **データ分析** | エージェントがPythonスクリプトを実行してデータを処理、グラフを生成 |
| **コード実行** | ユーザー提供のコードスニペットを実行し結果を返す |
| **ファイル操作** | ドキュメントの読み取り、編集、プロジェクトファイルの管理 |
| **MCPサーバー** | 分離された環境でMCPツールサービスを実行 |
| **自動化タスク** | 定期スクリプトの実行、システムメンテナンスタスク |

## バックエンドタイプ

LangBotは複数のサンドボックスバックエンドをサポートしています：

| バックエンド | 依存関係 | 特徴 | 使用例 |
|-------------|----------|------|--------|
| **Docker** | `docker` CLI | 完全なコンテナ分離、カスタムイメージ対応 | 本番環境、完全な分離が必要 |
| **Nsjail** | `nsjail` binary | 軽量Linuxサンドボックス、コンテナ不要 | Linuxサーバー、軽量デプロイ |
| **E2B** | `e2b` SDK | クラウドサンドボックスサービス | ローカルインフラ不要 |

<Tip>
`backend`設定で選択：`local`でDockerまたはNsjailを自動選択；`docker`/`nsjail`/`e2b`で特定のバックエンドを強制。
</Tip>

## 組み込みツール

サンドボックス機能はエージェントに6つの組み込みツールを提供します：

| ツール | 機能 | 例 |
|--------|------|-----|
| `exec` | サンドボックス内でシェルコマンドを実行 | `python script.py` |
| `read` | ワークスペースファイルの内容を読み取る | `/workspace/data.json`を読み取り |
| `write` | ファイルを作成または上書き | 新しい設定ファイルを書き込み |
| `edit` | 既存ファイルを編集（文字列置換） | コード内の関数を修正 |
| `glob` | パターンマッチングでファイルを検索 | `**/*.py`ファイルを検索 |
| `grep` | 正規表現でファイル内容を検索 | コード内でキーワードを検索 |

<Note>
- `exec`ツールはサンドボックスコンテナ内でコマンドを実行します；`read`/`write`/`edit`/`glob`/`grep`は`/workspace`にマッピングされたホストディレクトリを直接操作します。
- バックエンド（Docker/Nsjail/E2B）が1つも利用可能でない場合、これらのツールはエージェントに提供されません。
- これらのネイティブツールは組み込みの Local Agent 向けです。Dify、n8n、Langflow、Coze などの外部 Runner では、それぞれのプラットフォームのツール機構を使用します。
</Note>

## スコープとライフサイクル

### スコープ設定

パイプラインのAI設定で、**サンドボックススコープ**を設定できます。これはメッセージ間でサンドボックス環境をどのように共有するかを決定します：

| スコープ | テンプレート | 説明 |
|--------|------|------|
| グローバル | `{global}` | 全ユーザーが同じサンドボックスを共有 |
| チャットごと（推奨） | `{launcher_type}_{launcher_id}` | 同じチャット内のメッセージがサンドボックスを共有 |
| チャット内のユーザーごと | `{launcher_type}_{launcher_id}_{sender_id}` | 同じチャット内の異なるユーザーを分離 |
| 会話コンテキストごと | `{launcher_type}_{launcher_id}_{conversation_id}` | 会話コンテキストで分離 |
| メッセージごと | `{query_id}` | 各メッセージが完全に分離 |

### ライフサイクル

サンドボックスセッションのライフサイクルルール：

| 条件 | 動作 |
|------|------|
| アイドルタイムアウト | 5分間（300秒）使用されないと自動クリーンアップ |
| 実行中のプロセスがある | クリーンアップせず、プロセス終了を待機 |
| `persistent: true` 設定 | 自動クリーンアップなし |

<Warning>
同じセッション内のコマンドはファイルシステムの状態を共有します。完全な分離が必要な場合は「メッセージごと」スコープを使用してください。
</Warning>

## ワーキングディレクトリとファイルマウント

### ディレクトリ構造

各バックエンドのデフォルトワーキングディレクトリ：

| バックエンド | コンテナ内パス | 説明 |
|-------------|-----------|------|
| Docker | `/workspace` | ホストの`host_root`ディレクトリをマウント |
| Nsjail | `/workspace` | ホストの`host_root`ディレクトリをマウント |
| E2B | `/home/user/workspace` | クラウドサンドボックスの書き込み可能ディレクトリ |

エージェントが`exec`/`read`/`write`/`edit`/`glob`/`grep`ツールで操作するのはすべて`/workspace`ディレクトリ下のファイルです。E2Bバックエンドはパスを自動的に`/home/user/workspace`にマッピングします。

### マウント設定

`config.yaml`でホストマウントを設定：

```yaml
box:
  local:
    host_root: './data/box'           # ホスト上のワーキングディレクトリ
    skills_root: 'skills'             # Skill パッケージディレクトリ。デフォルトは <host_root>/skills
    allowed_mount_roots:              # エージェントがマウントできるディレクトリのホワイトリスト
      - './data/box'
      - '/tmp'
```

- `host_root`：デフォルトのワーキングディレクトリ、サンドボックス内の`/workspace`にマッピング
- `skills_root`：Box Runtime が管理する Skill パッケージディレクトリ。相対パスは `host_root` 配下に解決され、デフォルトは `<host_root>/skills`
- `allowed_mount_roots`：エージェントが追加マウントをリクエストできるディレクトリのホワイトリスト

## クイックスタート

1. Dockerがインストールされ実行されていることを確認（または他のバックエンドを選択）
2. `config.yaml`でサンドボックスを設定：

```yaml
box:
  backend: 'local'
  local:
    profile: 'default'
    host_root: './data/box'
    skills_root: 'skills'
```

3. LangBotを起動、サンドボックス機能は自動的に有効
4. パイプラインで Local Agent を選び、function calling をサポートするモデルを使用
5. エージェントは自動的に`exec`/`read`/`write`/`edit`/`glob`/`grep`ツールを取得

<Tip>
サンドボックス機能は自動的に有効になります。バックエンドが利用可能であれば、Local Agent は組み込みツールを取得します。パイプラインで sandbox を個別に有効化する必要はありません。
</Tip>
